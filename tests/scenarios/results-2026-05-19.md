## Anomaly detail — T1

**Original spec claim**: "Layer 1 regex deny runs at the Claude Code
permission layer **before reaching the AST hook**."

**Observed**: The deny reason text returned by the Bash tool came from
the AST hook (`[rm-recursive-force] (data-loss)`), not from the
permission check. Claude Code's documented behaviour is that
`PreToolUse` hooks execute before the permission check; if a hook
returns `deny`, the tool is blocked and the permission check never
runs.

**Why this is not a safety regression**:
- The command IS blocked. The firewall achieves its goal.
- L1 still independently covers non-Bash tools (`Read(.env)`, etc.)
  where the AST hook is not invoked.
- L1 is the fallback for the case where the AST hook crashes or fails
  open. Defence in depth holds.

**Spec correction applied**: `docs/firewall.md` now states that L2
fires first for Bash matchers, and L1 acts as the fallback. The
two-layer framing is preserved.
